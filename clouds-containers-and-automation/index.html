Why Automate?

Automation is the key to speed, consistency, scalability and repeatability.

Automation is about taking manual processes and placing technology around them to make them repeatable.

Immutable Infrastructure

  • Cattle vs pets
  • Snowflake Servers vs. Phoenix Servers

Immutable Architecture

Continuous delivery

  • Automated Pipeline from developer commit to production deploy
  • Requires Continuous Integration where every commit triggers all the tests
  • Requires lots of tests
  • Deploying more frequently lowers risk

Continuous delivery antipatterns

  • Deploying software manually
  • Manual configuration of production environments
  • Deploying to a production-like environment only after development is complete

MTTR vs. MTBF

  • MTTR = Mean Time To Repair
  • MTBF = Mean Time Between Failures
  • MTTD = Mean Time To Detection
  • A low MTTR requires a low MTTD
  • A low MTTR requires Automation
  • A low MTTD requires a good Monitoring

System Containers

  • System containers vs. application containers
  • LXC
  • OpenVZ
  • Namespaces and cgroups were written for system containers
  • Unified Kernel API
  • Docker came along and stole the thunder
  • Layers (UnionFS) and workflow

LXC/LXD

  • Written to be a drop-in replacement for KVM
  • 10x increase in number of instances on the same hardware
  • Faster to start all instances
  • Better disk and network IO (up to 40%)
  • Security ...

Summing up

  • Deep integration required particularity for networking and volumes
  • Running containers in VMs is common but not ideal for performance
  • People are running containers in VMs due to security concerns
  • Original container use cases are still valid and interesting (see LXC/LXD)
  • Orthogonal concerns (application vs machine abstractions) can make for difficult fit
  • So many ways to orchestrate things, expect consolidation
