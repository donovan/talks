What is the Three-Tier Architecture?

  1. Presentation Tier - A web browser on a workstation
  2. Application Tier - The business logic expressed as code
  3. Data Tier - A relational database

The Three-Tier Architecture

What is the Four-Tier Application Architecture?

Is the Three-Tier Architecture still useful? A blog post from Ted Schadler at Forrester Research suggests a new architecture described as “The Four-Tier Engagement Platform”.

What is the Four-Tier Application Architecture?

  • Client Tier
  • Delivery Tier
  • Aggregation Tier
  • Services Tier

The Four-Tier Application Architecture

Client Tier

  1. Mobile clients (native and responsive html)
  2. Web clients (mobile and desktop)
  3. Other clients (IOT etc.)
  4. Responsible for experience delivery

Client Tier Features

  1. Decoupled client and backend
  2. Backend supports different clients
  3. Single-page Applications
  4. Responsive
  5. Consumes services

Client Tier Drivers

  1. Regular refreshes of frontend components
  2. TODO

Client Tier Issues

  1. Javascript stratigraphy
  2. Javascript mediated SEO

Delivery Tier

  1. Optimizes content for proper display on device
  2. Caches content for performant delivery

Delivery Tier Features

  1. Fine grained cache control
  2. Anycasting
  3. Large number of global POPs
  4. NZ POPs
  5. New HTTP and SSL Features
  6. WAF

Delivery Tier Drivers

  1. DDoS
  2. Better latency
  3. Performance globally
  4. Performance optimisation
  5. Outsource client facing infrastructure challenges

Delivery Tier Issues

  1. Its a proxy!
  2. Its not under your control
  3. Its hard to build yourself
  4. Complexity (Routing, BGP, DNS etc.)
  5. The boundary of your service is global
  6. Peering
  7. Worse latency

Aggregation Tier

  1. Aggregates and federates services tier
  2. Provides discovery for underlying services
  3. Performs data protocol translation

Aggregation Tier Features

  1. Authentication
  2. SSL Termination
  3. Load Balancing
  4. ACLS, CORS, Rate limiting
  5. Analytics, Monitoring and Logging
  6. Transformation

Aggregation Tier Drivers

  1. Provide a gateway to services
  2. Access control and limiting
  3. Aggregation (including third party services)

Aggregation Tier Issues

  1. Protocol translation can lead you to an ESB
  2. Its a proxy!

Services Tier

  1. Existing on premises systems
  2. External third party services

Services Tier Features

  1. Mircoservice architecture
  2. Or a deprecated monolith ...

Services Tier Drivers

  1. Scale services not machines
  2. Decouple service delivery from the monolith
  3. Rewriting the monolith is required but difficult and costly
  4. Adding new features to the monolith is getting harder and harder due to layers of technical debt
  5. Useful services can be decoupled and delivered with reasonable budgets and timeframes

How do you move from a Monolithic to a Microservice architecture?

  1. Do you need to move?
  2. Microservices _will_ add complexity, at what point do they become worth it?
  3. Greenfields vs. existing site.
  4. Build or $SERVICEaaS

How do you move from a Monolithic to a Microservice architecture?

  1. New services become microservices
  2. Abstract existing services from the monolith over time

Rules of thumb

  1. New services become microservices
  2. Abstract existing services from the monolith over time
  3. "Choose Boring Technology" for the Monolith

Containers != Microservices

  1. Orthogonal
  2. Containers are a great way to build microservices
  3. Netflix defined the microservice pattern long before docker existed

How do achieve data consistency in a distributed system?

  1. You don't
  2. You need to embrace eventual consistency
  3. Embrace redundancy

CAP Theorem

Ripples

  1. Ripples (a propagating wave) moves from the source to the edges (clients)
  2. How can all services have the same view of the world?

Consistent keys not consistent data

  1. Need to be able to refer to the same piece of data across services in a consistent manner
  2. Natural keys
  3. Source and all nodes expose the keys
  4. NoSQL (key/value stores) help

Data injection

  1. New data added later on
  2. Don't couple it with a particular node

Split channels

  1. Different types of data have different latency and "currency"
  2. Dedicated pipelines
