Is the Three-Tier Architecture still useful? A blog post from Ted Schadler at Forrester Research suggests a new architecture described as “The Four-Tier Engagement Platform”.

What is the Four-Tier Application Architecture?

  1. Client Tier
  2. Delivery Tier
  3. Aggregation Tier
  4. Services Tier

The Four-Tier Application Architecture

Client Tier

  • Mobile clients (native and responsive html)
  • Web clients (mobile and desktop)
  • Other clients (IOT etc.)
  • Responsible for experience delivery

Client Tier Features

  • Decoupled client and backend
  • Backend supports different clients
  • Single-page Applications
  • Responsive
  • Consumes services

Client Tier Drivers

  • Regular refreshes of frontend components
  • TODO

Client Tier Issues

  • Javascript stratigraphy
  • Javascript mediated SEO

Delivery Tier

  • Optimizes content for proper display on device
  • Caches content for performant delivery

Delivery Tier Features

  • Fine grained cache control
  • Anycasting
  • Large number of global POPs
  • NZ POPs
  • New HTTP and SSL Features
  • WAF

Delivery Tier Drivers

  • DDoS
  • Better latency
  • Performance globally
  • Performance optimisation
  • Outsource client facing infrastructure challenges

Delivery Tier Issues

  • Its a proxy!
  • Its not under your control
  • Its hard to build yourself
  • Complexity (Routing, BGP, DNS etc.)
  • The boundary of your service is global
  • Peering
  • Worse latency

Aggregation Tier

  • Aggregates and federates services tier
  • Provides discovery for underlying services
  • Performs data protocol translation

Aggregation Tier Features

  • Authentication
  • SSL Termination
  • Load Balancing
  • ACLS, CORS, Rate limiting
  • Analytics, Monitoring and Logging
  • Transformation

Aggregation Tier Drivers

  • Provide a gateway to services
  • Access control and limiting
  • Aggregation (including third party services)

Aggregation Tier Issues

  • Protocol translation can lead you to an ESB
  • Its a proxy!

Services Tier

  • Existing on premises systems
  • External third party services

Services Tier Features

  • Mircoservice architecture
  • Or a deprecated monolith ...

Services Tier Drivers

  • Scale services not machines
  • Decouple service delivery from the monolith
  • Rewriting the monolith is required but difficult and costly
  • Adding new features to the monolith is getting harder and harder due to layers of technical debt
  • Useful services can be decoupled and delivered with reasonable budgets and timeframes

How do you move from a Monolithic to a Microservice architecture?

  • Do you need to move?
  • Microservices _will_ add complexity, at what point do they become worth it?
  • Greenfields vs. existing site.
  • Build or $SERVICEaaS

How do you move from a Monolithic to a Microservice architecture?

  • New services become microservices
  • Abstract existing services from the monolith over time

Rules of thumb

  • New services become microservices
  • Abstract existing services from the monolith over time
  • "Choose Boring Technology" for the Monolith

Containers != Microservices

  • Orthogonal
  • Containers are a great way to build microservices
  • Netflix defined the microservice pattern long before docker existed

How do achieve data consistency in a distributed system?

  • You don't
  • You need to embrace eventual consistency
  • Embrace redundancy

CAP Theorem

Ripples

  • Ripples (a propagating wave) moves from the source to the edges (clients)
  • How can all services have the same view of the world?

Consistent keys not consistent data

  • Need to be able to refer to the same piece of data across services in a consistent manner
  • Natural keys
  • Source and all nodes expose the keys
  • NoSQL (key/value stores) help

Data injection

  • New data added later on
  • Don't couple it with a particular node

Split channels

  • Different types of data have different latency and "currency"
  • Dedicated pipelines
